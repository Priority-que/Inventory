# Inventory 后端接口文档 - 前端联调版

- 更新时间：2026-04-26
- 文档范围：`inventory_back` 当前前端联调接口；其中 `/agent/workflow/execute` 由 Java 后端代理到 Python Agent 服务执行
- 覆盖规模：涉及 18 个前端联调 Controller，63 个前端联调接口；另有 `AgentContextController` 为 Agent 内部支撑，不计入前端联调接口
- 文档口径：严格以当前后端代码、DTO/VO、Mapper XML、ServiceImpl 校验逻辑为准，不按历史业务文档补写未实现接口
- Agent 口径：前端聊天流只保留 `GET /agent/session/list` 和 `POST /agent/workflow/execute`；旧 Java Agent 单点接口和 Agent 内部上下文接口不再作为前端联调接口

## 1. 接口总览

### 1.1 公共约定

- 基础地址：`http://{host}:8080`
- 默认请求头：
  - `Content-Type: application/json`
  - `Authorization: Bearer <token>`，除 `/auth/login` 外，其余接口都需要登录
- 文件上传接口使用 `multipart/form-data`
- JWT 令牌有效期：120 分钟
- Token 前缀：`Bearer`
- 当前代码层面主要做了“是否登录”控制，未对大多数业务接口增加显式 `@PreAuthorize` / `@Secured` / `@RolesAllowed`
- 前端联调时，请优先以响应体中的 `code` 判断业务成功与否
- 前端只需要请求 Java 后端 `8080`；Python Agent 服务 `8000` 是 Java 后端内部上游，不建议前端直接调用

### 1.2 响应与错误处理约定

#### 1.2.1 标准响应结构

除 `/api/oss/test/upload` 外，大多数接口统一返回：

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | `Integer` | 业务状态码。成功通常为 `200`，业务失败常见为 `401` / `403` / `500` |
| `msg` | `String` | 提示信息 |
| `data` | `Object` | 业务数据，可能是对象、数组、分页对象、字符串，或 `null` |

说明：

- `/agent/workflow/execute` 当前由 Java 代理 Python Agent 服务，正常成功时仍返回同样的 `{ code, msg, data }` 结构。
- 当 Python Agent 服务不可用时，Java 代理层会返回 HTTP `502`，响应体仍使用标准 `Result` 结构。

#### 1.2.2 分页响应真实结构

当前项目分页对象来自 MyBatis-Plus `Page`，实际序列化字段为：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "records": [],
    "total": 0,
    "size": 10,
    "current": 1,
    "pages": 0
  }
}
```

分页字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `records` | `Array` | 当前页数据列表 |
| `total` | `Long` | 总条数 |
| `size` | `Long` | 每页条数 |
| `current` | `Long` | 当前页码 |
| `pages` | `Long` | 总页数 |

#### 1.2.3 安全错误示例

未登录或 Token 失效时，Spring Security 会直接返回 HTTP `401`：

```json
{
  "code": 401,
  "msg": "未登录或者登录已失效",
  "data": null
}
```

无权限时，Spring Security 会直接返回 HTTP `403`：

```json
{
  "code": 403,
  "msg": "无权限访问",
  "data": null
}
```

业务异常或普通异常多数情况下返回 HTTP `200`，但响应体 `code` 为 `500`：

```json
{
  "code": 500,
  "msg": "系统异常，请联系管理员！",
  "data": null
}
```

#### 1.2.4 当前实现中的错误处理差异

- Spring Security 直接拦截的未登录/无权限，会返回真实 HTTP `401` / `403`
- Controller / Service 内部通过 `Result.error(...)` 返回的业务错误，通常 HTTP 状态仍为 `200`
- 因此前端不能只看 HTTP 状态，必须同时判断响应体中的 `code`

### 2.2 `POST /auth/login`

- 接口说明：账号密码登录，成功后返回 JWT Token
- 鉴权要求：否
- 推荐调用角色：全角色
- 请求头：`Content-Type: application/json`
- 请求 Body：`LoginDTO`
- 成功响应：`Result<LoginVO>`

请求示例：

```json
{
  "username": "admin001",
  "password": "123456"
}
```

成功示例：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9.xxx.yyy",
    "tokenType": "Bearer",
    "expiresIn": 7200,
    "userId": 1,
    "username": "admin001",
    "name": "系统管理员001",
    "roleCodes": ["ADMIN"]
  }
}
```

典型失败：

```json
{
  "code": 401,
  "msg": "用户名或密码错误",
  "data": null
}
```

实现备注：

- 登录成功后，后端会更新 `lastLoginTime`
- 当前代码只接受用户名密码，不支持验证码、短信、刷新 Token

### 2.3 `POST /auth/logout`

- 接口说明：退出登录，并将当前 Token 加入 Redis 黑名单
- 鉴权要求：是
- 推荐调用角色：全角色
- 请求头：`Authorization: Bearer <token>`
- 请求参数：无
- 成功响应：`Result<null>`

成功示例：

```json
{
  "code": 200,
  "msg": "退出登录成功",
  "data": null
}
```

典型失败：

```json
{
  "code": 401,
  "msg": "未登录或登录已失效",
  "data": null
}
```

实现备注：

- 该接口内部会再次从请求头中解析 Token
- 如果请求已通过 Security，但业务层又取不到合法 Token，仍可能返回 `code=401`

### 2.4 `GET /auth/me`

- 接口说明：获取当前登录用户信息
- 鉴权要求：是
- 推荐调用角色：全角色
- 请求头：`Authorization: Bearer <token>`
- Query 参数：无
- 成功响应：`Result<CurrentUserVO>`

成功示例：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "id": 1,
    "username": "admin001",
    "name": "系统管理员001",
    "dept": "平台运营",
    "status": "ENABLED",
    "roleCodes": ["ADMIN"]
  }
}
```

典型失败：未登录时返回 `401`

### 2.5 `PUT /auth/password`

- 接口说明：修改当前登录用户密码，成功后当前 Token 会失效
- 鉴权要求：是
- 推荐调用角色：全角色
- 请求头：`Authorization: Bearer <token>`
- 请求 Body：`ChangePasswordDTO`
- 成功响应：`Result<null>`

请求示例：

```json
{
  "oldPassword": "123456",
  "newPassword": "12345678",
  "confirmPassword": "12345678"
}
```

成功示例：

```json
{
  "code": 200,
  "msg": "修改密码成功",
  "data": null
}
```

典型失败：

```json
{
  "code": 500,
  "msg": "旧密码错误",
  "data": null
}
```

实现备注：

- 新旧密码一致校验当前没有单独限制
- 修改成功后会把当前 Token 拉黑，前端应主动跳回登录页

### 18.2 `POST /agent/workflow/execute`

- 接口说明：统一 Workflow Agent 入口。前端请求 Java 后端该接口，Java 后端会代理转发到 Python Agent 服务执行工作流
- 鉴权要求：是

- 请求 Body：`WorkflowAgentRequest`
- 成功响应：`Result<WorkflowAgentResponse>`

- 认证头：前端仍只需向 Java 传 `Authorization: Bearer <token>`；Java 会原样转发给 Python Agent

请求示例：

```json
{
  "message": "帮我分析 PO202604211230001001 为什么还没完成",
```

实现备注：

- 前端只调用 Java 后端 `8080` 的 `/agent/workflow/execute`，不要直接调用 Python Agent `8000`

- Java 代理层正常情况下会透传 Python 返回的 HTTP 状态、响应体和 `Content-Type`，不会再额外包一层 `Result.success(...)`
- 缺省 `threadId` 时，Python Agent 会自动生成 `agt-{uuid}`
- 接口会由 Python Agent 写入 `agent_session`、`agent_message`、`agent_session_state`、`agent_result`

### 20.2 当前实现注意事项

以下内容是按当前代码核对后发现的联调注意事项，前端开发时需要按实际接口处理，不要按历史需求文档假设已实现。

1. 当前后端没有独立的角色 CRUD Controller，只有用户绑定角色相关接口。
2. 当前后端没有供应商“提交审核、审核通过、审核驳回、停用”的专用接口，供应商模块目前只有分页、新增、修改、删除、附件上传。
3. 当前后端没有供应商确认采购订单接口，因此 `WAIT_CONFIRM -> IN_PROGRESS` 的正式接口尚未暴露。
4. 当前后端没有库存台账查询和库存流水查询 Controller，库存只在确认入库时被内部更新。
5. 大部分接口只有登录校验，没有显式角色权限注解，文档中的“推荐调用角色”是业务建议，不代表代码已强制。
6. 部分新增/修改接口缺少后端硬校验，例如物料、仓库、供应商新增/修改，前端需要按业务表单自行校验。
7. 部分业务错误通过 `Result.error(...)` 返回，HTTP 状态仍可能是 `200`，前端必须看响应体 `code`。
8. `/api/oss/test/upload` 返回原始 `Map`，没有 `Result` 包裹。
9. `SupplierServiceImpl.addSupplier` 插入失败时当前代码返回 `Result.success("添加供应商失败！")`，响应 `code` 仍为 `200`。
10. `PurchaseRequestServiceImpl.addPurchaseRequest`、`updatePurchaseRequest`、`deletePurchaseRequest` 的失败文案当前存在“成功”字样，前端应以 `code` 为准。
11. `PurchaseRequestPage` 的 `dept` 查询 SQL 当前使用了 `requestNo` 参数值，部门筛选可能不符合预期。
12. `WarehousePage` 的 `name` 查询条件判断当前依赖 `code` 是否为空，仓库名称筛选可能不符合预期。
13. 部分列表 XML 没有统一加 `deleted=0` 条件，联调数据如包含逻辑删除记录，需要后端修正 SQL。
14. `PurchaseOrderItem` 修改单价接口的错误文案是“不能小于0”，但代码实际要求 `unitPrice > 0`。
15. Agent 对外聊天流只保留 `/agent/session/list` 和 `/agent/workflow/execute`；旧 Java Agent 单点接口和 `/agent/context/*` 内部上下文接口不再作为前端联调入口。
16. `/agent/workflow/execute` 当前由 Java 后端代理到 Python Agent 服务执行；前端仍只请求 Java 后端，不需要直连 Python `8000`。
17. Python Agent 不可用时，Java 代理层会返回 HTTP `502` 和标准 `Result` 响应体，前端需要按上游不可用场景提示用户稍后重试。

### 20.3 前端联调建议

- 登录后统一保存 `tokenType + token`，请求头传 `Authorization: Bearer <token>`
- 所有列表页都按 `{ records, total, size, current, pages }` 解析分页
- 状态按钮由前端按枚举和业务规则控制，不能完全依赖后端当前权限
- 采购申请详情页需要分别调用申请详情和申请明细接口
- 采购订单详情页需要分别调用订单详情和订单明细接口
- 到货详情和入库详情接口本身会带 `items`，不需要再额外查询明细
- 文件上传统一使用 `multipart/form-data`，业务附件优先用 `/supplierFile/uploadSupplierFile`
- Workflow Agent 连续对话必须复用 `threadId`
- Workflow Agent 前端入口固定使用 Java 后端 `/agent/workflow/execute`，不要在浏览器侧配置 Python Agent 地址
