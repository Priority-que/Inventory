# 面向中小制造企业的供应商协同采购与仓储履约平台
# V1 统一系统设计文档

## 1. 文档目的

本文档用于给一期开发提供统一的技术设计基线，重点解决现有文档中的定义冲突，并明确后端分层、模块边界、数据库结构、状态模型和开发约束。

## 2. 统一设计决策

为保证系统可直接进入开发阶段，一期统一采用如下决策：

- 架构形态：模块化单体应用，不拆微服务。
- 权限模型：采用标准 RBAC 表结构，初始化时内置五类角色。
- 业务角色：管理员、采购员、采购主管、仓库岗、供应商。
- 审批模型：单级审批。
- 订单模型：一张采购申请只生成一张采购订单。
- 订单状态：去掉容易产生歧义的 `CONFIRMED`，供应商确认后订单直接进入 `IN_PROGRESS`。
- 到货状态：到货单仅保留 `NORMAL/ABNORMAL`，订单累计履约进度通过采购订单状态表达。
- 入库状态：统一为 `PENDING/COMPLETED/CANCELLED/ABNORMAL`。
- 库存预警：动态计算，不新增独立预警状态表。
- 退货管理：不纳入一期，待二期扩展。

## 3. 总体架构

### 3.1 架构说明

系统采用前后端分离架构：

- 前端：后台管理端和供应商端
- 后端：单个 Spring Boot 应用，按业务模块组织
- 数据库：MySQL 8.0
- 缓存：Redis，可按需接入
- 文件存储：本地存储或 MinIO，按部署环境决定

### 3.2 模块化单体原因

对于两人团队和一期范围，模块化单体优于微服务：

- 业务链路强耦合，拆服务收益低
- 一期重点在于闭环业务而非分布式治理
- 开发、联调、部署、排错都更简单
- 对简历项目来说，清晰业务分层比空壳微服务更有说服力

## 5. 模块划分

建议后端按以下模块组织：

| 模块 | 主要职责 | 核心表 |
|---|---|---|
| `auth` | 登录、鉴权、当前用户信息 | `sys_user`、`sys_role`、`sys_user_role` |
| `system` | 用户、角色、菜单、字典、系统参数 | `sys_role`、`sys_menu`、`sys_role_menu`、`sys_dict_type`、`sys_dict_data`、`sys_config` |
| `base` | 物料、仓库基础数据 | `base_material`、`base_warehouse` |
| `supplier` | 供应商档案、审核 | `supplier` |
| `purchase` | 采购申请、审批、采购订单 | `purchase_request`、`purchase_request_item`、`purchase_request_approval`、`purchase_order`、`purchase_order_item` |
| `warehouse` | 到货、入库 | `arrival_record`、`arrival_record_item`、`inbound_order`、`inbound_order_item` |
| `inventory` | 库存台账、库存流水、预警查询 | `inventory`、`inventory_flow` |
| `message` | 消息中心 | `sys_message` |
| `log` | 操作日志 | `sys_oper_log` |
| `report` | 统计报表 | 聚合查询，无专用表 |

## 7. 认证与权限设计

### 7.1 用户模型

- `sys_user` 存放登录账号
- `sys_role` 存放角色
- `sys_user_role` 存放用户与角色关联
- `sys_menu` 存放菜单和按钮权限
- `sys_role_menu` 存放角色与菜单关联

### 7.2 初始角色

- `ADMIN`
- `PURCHASER`
- `PURCHASE_MANAGER`
- `WAREHOUSE`
- `SUPPLIER`

### 7.3 权限控制策略

- 菜单显隐基于角色与菜单关联
- 接口鉴权基于权限标识或角色编码
- 供应商数据接口必须额外校验“当前用户是否为所属供应商”

## 8. 统一状态模型

### 8.1 供应商状态

| 状态 | 说明 |
|---|---|
| `DRAFT` | 资料未完善，未提交 |
| `PENDING` | 已提交待审核 |
| `REJECTED` | 审核驳回 |
| `ACTIVE` | 审核通过，可合作 |
| `DISABLED` | 已停用 |

### 8.2 采购申请状态

| 状态 | 说明 |
|---|---|
| `DRAFT` | 草稿 |
| `PENDING_APPROVAL` | 待审批 |
| `APPROVED` | 审批通过 |
| `REJECTED` | 审批驳回 |
| `WITHDRAWN` | 已撤回 |
| `ORDER_CREATED` | 已生成采购订单 |

### 8.3 采购订单状态

| 状态 | 说明 |
|---|---|
| `WAIT_CONFIRM` | 待供应商确认 |
| `IN_PROGRESS` | 供应商已确认，订单履约中 |
| `PARTIAL_ARRIVAL` | 已发生部分到货或部分入库 |
| `COMPLETED` | 全部完成入库 |
| `CLOSED` | 人工关闭 |
| `CANCELLED` | 待确认阶段取消 |

### 8.4 到货状态

| 状态 | 说明 |
|---|---|
| `NORMAL` | 本次到货正常 |
| `ABNORMAL` | 本次到货存在异常 |

### 8.5 入库单状态

| 状态 | 说明 |
|---|---|
| `PENDING` | 待入库 |
| `COMPLETED` | 已完成入库 |
| `CANCELLED` | 已取消 |
| `ABNORMAL` | 异常待处理 |

## 9. 核心业务流程设计

### 9.1 供应商审核

1. 管理员创建供应商账号。
2. 供应商完善资料，状态从 `DRAFT` 变更为 `PENDING`。
3. 管理员审核通过后更新为 `ACTIVE`，审核驳回则更新为 `REJECTED`。

### 9.2 采购申请

1. 采购员创建采购申请主单和明细，初始状态 `DRAFT`。
2. 提交审批后状态变更为 `PENDING_APPROVAL`，同时写审批历史和待办消息。
3. 审批通过后更新为 `APPROVED`，驳回则更新为 `REJECTED`。

### 9.3 采购订单

1. 基于 `APPROVED` 状态采购申请生成采购订单。
2. 创建采购订单后，采购申请回写为 `ORDER_CREATED`。
3. 采购订单初始状态为 `WAIT_CONFIRM`。
4. 供应商确认订单后，更新供应商反馈交期、确认时间，订单状态更新为 `IN_PROGRESS`。

### 9.4 到货登记

1. 仓库岗选择 `IN_PROGRESS` 或 `PARTIAL_ARRIVAL` 的订单进行到货登记。
2. 到货主表记录本次到货总体信息。
3. 到货明细表记录本次各物料到货数量、合格数量、不合格数量。
4. 若累计仍未全部完成，订单状态维持或更新为 `PARTIAL_ARRIVAL`。

### 9.5 入库确认

1. 仓库岗基于到货记录创建入库单，状态初始为 `PENDING`。
2. 确认入库后，入库单更新为 `COMPLETED`。
3. 系统按物料与仓库维度更新库存台账。
4. 同时写入库存流水。
5. 若订单全部明细已完成入库，则订单状态更新为 `COMPLETED`。

## 10. 数据库设计总览

### 10.1 设计原则

- 业务表以逻辑外键为主，不强制建立物理外键。
- 统一使用 `BIGINT` 作为主键类型。
- 状态字段统一使用 `VARCHAR(32)`。
- 数量字段使用 `DECIMAL(18,3)`。
- 金额字段使用 `DECIMAL(18,2)`。
- 业务主表保留创建人、创建时间、修改人、修改时间、逻辑删除字段。
- 历史表和流水表以追加写入为主，不做逻辑删除。

### 10.2 表清单

| 分类 | 表名 |
|---|---|
| 权限与系统 | `sys_user`、`sys_role`、`sys_user_role`、`sys_menu`、`sys_role_menu`、`sys_dict_type`、`sys_dict_data`、`sys_config` |
| 基础数据 | `base_material`、`base_warehouse` |
| 供应商 | `supplier` |
| 采购 | `purchase_request`、`purchase_request_item`、`purchase_request_approval`、`purchase_order`、`purchase_order_item` |
| 仓储 | `arrival_record`、`arrival_record_item`、`inbound_order`、`inbound_order_item` |
| 库存 | `inventory`、`inventory_flow` |
| 消息与日志 | `sys_message`、`sys_oper_log` |

### 10.3 关键关系

- `supplier.user_id -> sys_user.id`
- `sys_user_role.user_id -> sys_user.id`
- `sys_user_role.role_id -> sys_role.id`
- `sys_role_menu.role_id -> sys_role.id`
- `sys_role_menu.menu_id -> sys_menu.id`
- `purchase_request.requester_id -> sys_user.id`
- `purchase_request_item.request_id -> purchase_request.id`
- `purchase_order.request_id -> purchase_request.id`
- `purchase_order.supplier_id -> supplier.id`
- `purchase_order_item.order_id -> purchase_order.id`
- `arrival_record.order_id -> purchase_order.id`
- `arrival_record_item.arrival_id -> arrival_record.id`
- `inbound_order.arrival_id -> arrival_record.id`
- `inbound_order_item.inbound_id -> inbound_order.id`
- `inventory.material_id -> base_material.id`
- `inventory.warehouse_id -> base_warehouse.id`
- `inventory_flow.inventory_id -> inventory.id`

## 11. 关键字段设计说明

### 11.1 快照字段

以下明细表需保留业务快照字段，避免基础数据变更影响历史单据：

- `purchase_request_item`
- `purchase_order_item`
- `arrival_record_item`
- `inbound_order_item`

快照字段包括：

- 物料编码
- 物料名称
- 规格型号
- 单位

### 11.2 业务编号

建议为以下对象生成唯一业务编号：

- 采购申请：`PRyyyyMMddNNNN`
- 采购订单：`POyyyyMMddNNNN`
- 到货单：`ARyyyyMMddNNNN`
- 入库单：`INyyyyMMddNNNN`
- 库存流水：`IFyyyyMMddNNNN`

### 11.3 库存台账唯一键

`inventory` 表需建立 `(material_id, warehouse_id)` 唯一索引，确保同一物料在同一仓库只有一条库存台账。

## 12. API 设计约束

- 列表接口统一支持分页与常用筛选条件。
- 状态变更接口采用明确动词，例如：`submit`、`approve`、`reject`、`confirm`、`cancel`。
- 同一接口要做状态前置校验，防止重复操作。
- 统一返回结构，包含 `code`、`message`、`data`。
- 文件上传统一走单独接口，业务表只保存文件地址。

## 13. 事务与一致性约束

- 采购申请提交、审批、生成采购订单均为本地事务。
- 供应商确认订单为本地事务。
- 到货登记为本地事务。
- 确认入库必须在一个事务内完成：入库单更新、库存台账更新、库存流水写入、订单状态重算。
- 一期不引入 Seata，不引入分布式事务。
- 消息通知和操作日志可以先采用本地事务内写表，后续如有性能需要再异步化。

## 14. 查询与 SQL 约束

- 简单单表 CRUD 允许使用 MyBatis-Plus 的 `BaseMapper`。
- 复杂列表查询、统计报表、聚合计算统一写在 MyBatis XML 中。
- Controller 禁止直接编写业务判断。
- Service 禁止拼装复杂 SQL。

## 15. Git 协作建议

建议以仓库根目录作为单一 Git 仓库，包含文档、后端和前端。

分支策略建议如下：

- `main`：稳定分支
- `feature/*`：功能分支
- `fix/*`：缺陷修复分支

两人协作建议按模块认领：

- 成员 A：认证、系统管理、基础数据、供应商模块
- 成员 B：采购、仓储、库存、消息、日志、报表模块

公共文件如状态枚举、统一响应、全局异常、数据库脚本，合并前应双方共同确认。

## 16. 测试建议

- 采购申请状态流转测试
- 采购审批幂等测试
- 采购订单生成前置条件测试
- 供应商只能查看本人订单的权限测试
- 到货数量越界校验测试
- 入库确认后库存台账和库存流水联动测试

## 17. 开发前检查清单

- 需求、状态、表结构三者是否一致
- SQL 建表脚本是否可重复执行
- 业务编号规则是否统一
- 初始角色和权限是否梳理完成
- 接口命名和分页返回格式是否统一
- 需要联调的主链路是否已确认顺序
