## 2. 一期设计原则

### 2.1 设计目标

一期必须完成以下能力：

- 用户登录
- 供应商资料维护与审核
- 采购申请
- 采购审批及审批历史
- 采购订单
- 供应商确认订单
- 到货登记
- 入库确认
- 库存台账
- 库存流水
- 消息中心
- 操作日志

### 2.2 外键策略

本项目所有表关联均采用 **虚拟外键** 设计：

- 数据库中仅保存关联 ID
- 不建立物理外键约束
- 由业务层校验关联对象是否存在
- 通过索引保证关联查询性能

例如：

- `supplier.user_id` 逻辑关联 `sys_user.id`
- `purchase_request.requester_id` 逻辑关联 `sys_user.id`
- `purchase_order.supplier_id` 逻辑关联 `supplier.id`
- `arrival_record.order_id` 逻辑关联 `purchase_order.id`

### 2.3 用户模型原则

一期采用 **单账号单岗位** 模型：

- 一个用户只对应一个岗位
- 不拆角色表和用户角色关系表
- 使用 `sys_user.post_type` 区分岗位

岗位类型如下：

- `ADMIN`：管理员
- `PURCHASER`：采购员
- `PURCHASE_MANAGER`：采购主管
- `WAREHOUSE`：仓库岗
- `SUPPLIER`：供应商账号

### 2.4 通用字段原则

除特殊表外，业务表统一包含：

| 字段名 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT` | 主键 ID |
| `create_by` | `BIGINT` | 创建人 ID |
| `create_time` | `DATETIME` | 创建时间 |
| `update_by` | `BIGINT` | 修改人 ID |
| `update_time` | `DATETIME` | 修改时间 |
| `deleted` | `TINYINT(1)` | 逻辑删除标记，`0` 未删除，`1` 已删除 |

### 2.5 类型规范

| 字段用途 | 建议类型 |
|---|---|
| 主键 ID | `BIGINT` |
| 状态字段 | `VARCHAR(32)` |
| 数量字段 | `DECIMAL(18,3)` |
| 金额字段 | `DECIMAL(18,2)` |
| 日期字段 | `DATE` |
| 时间字段 | `DATETIME` |
| 短文本 | `VARCHAR(64)` / `VARCHAR(128)` / `VARCHAR(255)` |
| 长文本 | `VARCHAR(500)` |

## 7.2 业务总流程

```mermaid
flowchart LR
    A["管理员准备基础数据"] --> B["供应商维护资料"]
    B --> C["管理员审核供应商"]
    C --> D["采购员创建采购申请"]

    K --> L["系统更新库存台账和库存流水"]
    L --> M["系统发送业务消息"]
    M --> N["管理员/采购岗/仓库岗查看业务结果"]
```

读取：

- `sys_message`

- `sys_oper_log`
- `operation_type = CONFIRM_INBOUND`

### 核心规则

- 入库数量不能超过合格数量
- 只有确认入库后库存才增加
- 库存变化必须落库存流水
- 入库后如果订单全部完成，则订单状态改为 `COMPLETED`

- 管理员

### 业务动作

- 按模块、操作人、时间查询日志
- 查看异常操作

### 涉及表

- `sys_oper_log`

### 表操作说明

#### 1. 查询日志
读取：
- `sys_oper_log`

常用维度：
- `log_type`
- `module_name`
- `operator_id`
- `operation_type`
- `operate_time`

### 核心规则

- 一期登录和业务操作统一写入这张表
- 后续如果日志量增大，再拆登录日志和业务日志

## 7.14 阶段十二：库存查询与预警

### 参与角色

- 管理员
- 采购岗
- 仓库岗

### 业务动作

- 查询库存台账
- 查询库存流水
- 判断低库存和超储

### 涉及表

- `inventory`
- `inventory_flow`
- `base_material`
- `base_warehouse`
- `sys_message`

### 表操作说明

#### 1. 查询库存台账
读取：
- `inventory`
- `base_material`
- `base_warehouse`

#### 2. 查询库存流水
读取：
- `inventory_flow`
- `inventory`
- `base_material`
- `base_warehouse`

#### 3. 动态计算预警
逻辑：
- `inventory.current_qty < base_material.safety_stock` => 低库存
- `inventory.current_qty > base_material.upper_limit_stock` => 超储

#### 4. 触发预警消息
新增：
- `sys_message`

接收人：
- 管理员
- 采购岗
- 仓库岗

类型：
- `INVENTORY_ALERT`

### 核心规则

- 预警结果一期可不单独落表
- 直接通过库存和物料阈值动态计算

## 9. 核心状态流转

### 9.1 供应商状态流转

```text
PENDING -> ACTIVE
PENDING -> REJECTED
ACTIVE -> DISABLED
REJECTED -> PENDING
```

### 9.2 采购申请状态流转

```text
DRAFT -> PENDING_APPROVAL
PENDING_APPROVAL -> APPROVED
PENDING_APPROVAL -> REJECTED
PENDING_APPROVAL -> WITHDRAWN
REJECTED -> DRAFT
APPROVED -> ORDER_CREATED
```

### 9.3 采购订单状态流转

```text
WAIT_CONFIRM -> CONFIRMED
CONFIRMED -> IN_PROGRESS
IN_PROGRESS -> PARTIAL_ARRIVAL
PARTIAL_ARRIVAL -> COMPLETED
IN_PROGRESS -> CLOSED
PARTIAL_ARRIVAL -> CLOSED
WAIT_CONFIRM -> CANCELLED
```

### 9.4 入库状态流转

```text
PENDING -> COMPLETED
PENDING -> CANCELLED
PENDING -> ABNORMAL
```

## 10. 一期设计结论

本方案采用 17 张表，能够完整支撑一期全部核心功能：

- 采购申请
- 审批流及审批历史
- 采购订单
- 供应商协同
- 到货登记
- 入库确认
- 库存台账
- 库存流水
- 消息中心
- 操作日志

并且仍然保持了以下特点：

- 结构完整
- 复杂度可控
- 不依赖物理外键
- 适合两人协作开发
- 后续可继续扩展

这份文档可以作为你们一期正式开发的数据库与业务基线版本。
