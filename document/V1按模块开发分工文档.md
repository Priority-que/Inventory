# 面向中小制造企业的供应商协同采购与仓储履约平台
# V1 按模块开发分工文档

## 1. 文档目标

本文档用于指导两人小组按模块开工，目标是：

- 把技术难、事务重、状态流转复杂的模块集中给技术更强的一侧
- 把 CRUD 为主、耦合低、便于并行推进的模块集中给技术相对弱的一侧
- 明确每个模块的功能范围、前置依赖、交付标准
- 降低两人同时修改同一批文件的概率，减少 Git 冲突

本文档基于以下基线文档制定：

- [V1统一需求说明书](D:/code/project/inventory/document/V1统一需求说明书.md)
- [V1统一系统设计文档](D:/code/project/inventory/document/V1统一系统设计文档.md)
- [V1__init.sql](D:/code/project/inventory/inventory_back/sql/V1__init.sql)

## 2. 分工原则

### 2.1 技术更强的一侧负责

- 认证鉴权
- 角色权限底座
- 状态机和事务边界明显的核心业务
- 采购、入库、库存这些强耦合主链路
- 通用基础设施和跨模块能力

### 2.2 技术相对弱的一侧负责

- 基础资料类 CRUD
- 配置类管理模块
- 查询类和展示类模块
- 供应商资料维护这种规则相对简单的模块
- 后续依赖主链路结果的读接口和报表接口

### 2.3 共同约定

- 公共代码只由一侧主改，另一侧谨慎修改。
- 同一个业务模块尽量只由一个人负责到底。
- DTO、VO、枚举、接口返回结构、异常码要统一。
- 简单单表 CRUD 用 MyBatis-Plus，复杂查询和报表统一写 XML。

## 3. 建议人员分组

### A 线：核心高难模块负责人

适合技术更强的同学，负责主链路和框架底座。

### B 线：基础低耦合模块负责人

适合技术相对弱的同学，负责基础管理和查询展示模块。

## 4. 开工顺序总览

### 阶段 0：先完成公共底座

由 A 线主导，B 线配合验证：

1. 初始化后端工程
2. 接入 MySQL
3. 落地统一返回结构和全局异常
4. 跑通 JWT 登录
5. 跑通 RBAC 基础鉴权
6. 导入建库脚本

只有这一步完成后，双方才适合正式并行开发。

### 阶段 1：第一次并行开发

- A 线：采购申请、审批、采购订单
- B 线：系统管理、基础数据、供应商资料

### 阶段 2：第二次并行开发

- A 线：到货、入库、库存、日志写入
- B 线：消息中心查询、报表查询、管理类补充接口

### 阶段 3：联调与收尾

- 采购主链路联调
- 权限联调
- 库存正确性联调
- 接口文档、测试数据、README、演示脚本整理

## 5. A 线模块清单

## A0 公共技术底座

### 模块定位

这是全项目最先做、也最重要的底座模块。

### 功能需求

- Spring Boot 工程初始化
- Maven 依赖管理
- 数据库连接与多环境配置
- 统一返回结构
- 全局异常处理
- 分页基础能力
- 通用枚举和常量
- 当前登录用户上下文
- Swagger 或 Knife4j 接口文档
- 文件上传基础能力

### 技术点

- 项目分层和包结构设计
- JWT 令牌签发与校验
- Spring Security 或等价权限方案
- 全局异常和统一响应封装
- 公共拦截器或过滤器

### 前置依赖

- 无

### 交付标准

- 项目可以启动
- 数据库可以连接
- 登录接口可用
- 受保护接口能正确拦截未登录请求
- Swagger 能看到基础接口

## A1 认证与 RBAC 底座

### 模块定位

这是所有模块的权限前提，必须由 A 线负责。

### 功能需求

- 登录
- 退出登录
- 当前用户信息查询
- 当前用户菜单查询
- 修改密码
- 基于角色的接口权限控制
- 基于角色的菜单权限控制

### 技术点

- 用户、角色、菜单、用户角色、角色菜单关联查询
- 登录态解析
- 自定义权限注解或权限判断逻辑
- 供应商只能访问自身数据的额外数据权限校验

### 前置依赖

- A0 完成

### 交付标准

- 管理员、采购员、采购主管、仓库岗、供应商五类角色能正确登录
- 不同角色返回不同菜单
- 无权限接口返回统一错误

## A2 采购申请与采购审批

### 模块定位

这是业务主链路的第一段，状态流转明显，耦合度高。

### 功能需求

- 新建采购申请
- 编辑采购申请
- 删除草稿采购申请
- 提交采购申请
- 撤回采购申请
- 采购申请列表
- 采购申请详情
- 待审批列表
- 审批通过
- 审批驳回
- 审批历史查询

### 技术点

- 采购申请状态流转
- 采购申请主表和明细表事务一致性
- 审批历史记录写入
- 重复提交、重复审批的幂等控制
- 角色权限控制

### 前置依赖

- A0 完成
- A1 完成
- B2 物料基础数据可用

### 交付标准

- 草稿、待审批、已通过、已驳回、已撤回、已转采购单状态流转正确
- 审批历史记录完整
- 同一张待审批单不能被重复审批

## A3 采购订单与供应商确认

### 模块定位

这是采购主链路第二段，承接申请审批结果，开始进入履约阶段。

### 功能需求

- 基于采购申请生成采购订单
- 采购订单列表
- 采购订单详情
- 取消采购订单
- 关闭采购订单
- 供应商查看我的订单
- 供应商确认订单
- 供应商反馈预计交期

### 技术点

- 采购申请转采购订单的事务处理
- 一张申请只能生成一张订单的唯一性控制
- 订单金额汇总
- 供应商数据隔离
- 订单状态流转控制

### 前置依赖

- A2 完成
- B4 供应商模块可用

### 交付标准

- 已审批通过申请才能生成订单
- 非 `ACTIVE` 供应商不能被选择
- 供应商确认后订单状态变为 `IN_PROGRESS`
- 待确认订单可取消，执行中订单可关闭

## A4 到货、入库、库存

### 模块定位

这是整个系统最重的事务模块，也是最能体现后端能力的模块。

### 功能需求

- 到货登记
- 到货列表
- 到货详情
- 基于到货记录创建入库单
- 确认入库
- 取消待入库单
- 入库单列表
- 入库单详情
- 库存台账查询
- 库存流水查询
- 库存预警查询

### 技术点

- 到货数量、合格数量、不合格数量校验
- 入库确认事务
- 库存台账新增或累加
- 库存流水记录
- 订单累计入库后状态重算
- 状态前置校验和并发保护

### 前置依赖

- A3 完成
- B3 仓库基础数据可用

### 交付标准

- 到货数量不能超过订单剩余数量
- 入库数量不能超过合格数量
- 入库确认后库存台账和库存流水同步更新
- 全部完成入库后订单状态变为 `COMPLETED`

## A5 操作日志写入与消息写入机制

### 模块定位

这是跨模块能力，建议由 A 线统一实现，B 线只负责调用。

### 功能需求

- 关键业务操作日志写入
- 审批待办消息写入
- 审批结果消息写入
- 订单待确认消息写入
- 到货消息写入
- 入库消息写入
- 库存预警消息写入

### 技术点

- 通用日志记录入口

### 前置依赖

## 10. 联调阶段重点

联调时优先验证下面这条主链路：

`供应商审核通过 -> 创建采购申请 -> 提交审批 -> 审批通过 -> 生成采购订单 -> 供应商确认 -> 到货登记 -> 确认入库 -> 查询库存`

联调阶段必须重点检查：

- 不同角色是否拿到正确菜单和接口权限
- 单据状态是否按设计流转
- 重复点击是否导致重复提交、重复审批、重复入库
- 入库后库存台账和库存流水是否一致
- 消息和日志是否按关键动作落表

## 11. 开工建议

如果你们想降低一开始的压力，最稳的方式是：

1. A 线先把底座和登录跑起来
2. B 线先拿字典、系统参数、物料、仓库这些 CRUD 建信心
3. A 线完成采购主链路上半段
4. B 线完成供应商和系统管理
5. A 线完成仓储和库存
6. 最后一起联调、补日志、补报表、补文档

这样拆分的好处是：

- A 线负责最容易出 bug 的地方
- B 线负责最容易稳定推进的地方
- 两边在大多数时间不会改同一批文件
- Git 冲突和联调压力都更小
