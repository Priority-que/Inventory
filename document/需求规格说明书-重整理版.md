## 一、项目背景

本系统面向中小制造企业的采购协同与仓储履约场景，围绕物料采购、供应商协作、到货入库和库存更新进行数字化管理。系统通过前端 Vue 管理端和 Spring Boot 后端服务，统一管理用户角色、物料、仓库、供应商、采购申请、采购订单、到货记录、入库单和库存流水等数据，减少线下申请审批、手工记录和跨岗位沟通带来的信息断点。

项目当前存在 AI 智能助手相关模块和 Python 工作流服务，但本文档优先梳理主业务闭环。AI/agent 能力可作为采购诊断、风险预警和知识问答的扩展能力，不作为采购入库主流程的前置条件。

### 1.1 建设目标

- 为采购员提供采购申请创建、申请明细维护、提交审批、采购订单生成、订单执行跟踪、到货和入库结果查看等能力。
- 为采购经理提供待审批采购申请查询、申请明细查看、审批通过、审批驳回和审批历史追溯能力。
- 为仓库岗提供到货登记、异常到货记录、入库单生成、入库确认、入库取消和库存变更触发能力。
- 为系统管理员提供用户、角色、物料、仓库、供应商、供应商附件和操作日志等基础管理能力。
- 为供应商角色提供供应商资料维护和附件上传能力，并为后续订单确认、反馈交期等协同功能预留数据字段。
- 通过单据状态和库存流水保证采购申请、采购订单、到货、入库、库存之间的数据可追踪。

### 1.2 用户角色

| 角色 | 主要职责 |
| --- | --- |
| 系统管理员 | 维护用户、角色、物料、仓库、供应商档案和操作日志，查看平台整体数据。 |
| 采购员 | 创建采购申请，维护申请明细，提交审批，基于已审批申请生成采购订单，维护待确认订单信息，跟踪到货和入库结果。 |
| 采购经理 | 查询待审批采购申请，查看申请明细和审批历史，对采购申请执行审批通过或驳回。 |
| 仓库岗 | 查看采购订单，登记到货数量和合格/不合格数量，生成入库单，确认入库并触发库存台账和库存流水更新。 |
| 供应商 | 维护自身供应商资料和资质附件；订单确认和交期反馈为设计链路中的协同步骤，当前代码侧尚未完全形成独立接口。 |

### 1.3 系统范围

本系统需求范围覆盖以下模块：

- 认证与登录：登录、登出、当前用户信息、修改密码、JWT 鉴权和路由权限控制。
- 系统管理：用户管理、角色分配、用户状态维护、物料管理、仓库管理、操作日志查询。
- 供应商管理：供应商档案维护、供应商状态维护、资质附件上传和附件轮次管理。
- 采购管理：采购申请、采购申请明细、提交审批、撤回、审批通过/驳回、审批历史、采购订单、订单明细和订单关闭/取消。
- 仓储履约：到货登记、异常到货记录、入库单生成、入库取消、确认入库、库存台账更新、库存流水写入。
- 业务查询：采购员和仓库岗可按角色查看订单、到货、入库、物料、仓库等数据。
- 扩展能力：AI 智能助手、知识库、订单诊断、风险预警和供应商评分不纳入本文主链，但可使用主业务数据做辅助分析。

### 1.4 核心业务流程

系统主流程为：

```text
基础资料维护 -> 采购申请 -> 采购审批 -> 采购订单 -> 供应商确认 -> 到货登记 -> 入库确认 -> 库存更新
```

当前代码中已落地的强主链为：

```text
物料/仓库/供应商维护
-> 采购员创建采购申请和申请明细
-> 采购员提交审批
-> 采购经理审批通过或驳回
-> 采购员根据已通过申请生成采购订单
-> 仓库岗基于执行中的采购订单登记到货
-> 仓库岗根据到货单生成入库单
-> 仓库岗确认入库
-> 系统更新采购订单入库数量、库存台账和库存流水
```

需要特别注意：数据库和状态设计中存在 `WAIT_CONFIRM`、`IN_PROGRESS`、`supplier_date`、`confirm_time`、`supplier_note`，说明设计目标包含供应商确认订单。但当前前后端尚未提供明确的“供应商确认订单”接口和页面按钮；而到货登记要求订单状态必须为 `IN_PROGRESS` 或 `PARTIAL_ARRIVAL`。因此，`WAIT_CONFIRM -> IN_PROGRESS` 是当前主链中需要补齐的关键状态跳转。

## 二、整体用例说明

### 2.1 总体用例

系统围绕管理员、采购员、采购经理、仓库岗和供应商五类参与者展开。

- 管理员负责账号权限和基础资料，保证业务单据可以选择有效物料、仓库和供应商。
- 采购员负责采购需求提出和订单创建，是采购流程的发起者。
- 采购经理负责采购申请审批，是采购需求进入正式订单前的控制节点。
- 仓库岗负责收货、验收和入库，是库存变更的实际触发者。
- 供应商负责资料维护，并在设计目标中承担订单确认和交期反馈。

### 2.2 角色与页面入口

| 角色 | 主要页面 |
| --- | --- |
| ADMIN | 工作台、用户管理、物料管理、仓库管理、供应商管理、AI 智能助手、知识库管理、操作日志 |
| PURCHASER | 采购申请、采购订单、供应商查询、物料查询、到货跟踪、入库跟踪、AI 智能助手 |
| PURCHASE_MANAGER | 采购审批、AI 智能助手 |
| WAREHOUSE | 到货管理、入库管理、采购订单查看、物料查询、仓库查询、AI 智能助手 |
| SUPPLIER | 供应商资料维护、AI 智能助手 |

### 2.3 核心对象状态

| 业务对象 | 状态 |
| --- | --- |
| 用户/角色/物料/仓库 | `ENABLED`、`DISABLED` |
| 供应商 | `DRAFT`、`PENDING`、`REJECTED`、`ACTIVE`、`DISABLED` |
| 采购申请 | `DRAFT`、`PENDING_APPROVAL`、`APPROVED`、`REJECTED`、`WITHDRAWN`、`ORDER_CREATED` |
| 审批历史动作 | `SUBMIT`、`RESUBMIT`、`WITHDRAW`、`APPROVE`、`REJECT` |
| 采购订单 | `WAIT_CONFIRM`、`IN_PROGRESS`、`PARTIAL_ARRIVAL`、`COMPLETED`、`CLOSED`、`CANCELLED` |
| 到货单 | `NORMAL`、`ABNORMAL` |
| 入库单 | `PENDING`、`COMPLETED`、`CANCELLED`、`ABNORMAL` |
| 库存流水业务类型 | `INBOUND`、`INIT` |

## 三、用例描述

本章选取系统中的核心业务用例进行描述。每个用例从参与者、前置条件、后置条件、基本流程、异常流程和业务规则六个方面说明，便于后续开发、测试和答辩时说明系统关系。

### 3.1 注册登录/身份认证

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 注册登录/身份认证 |
| 参与者 | 管理员、采购员、采购经理、仓库岗、供应商 |
| 前置条件 | 用户账号已存在，后端认证服务和数据库可访问。 |
| 后置条件 | 登录成功后前端保存 token 和当前用户信息；登出后 token 失效；访问受保护路由时按角色控制。 |

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 供应商档案与附件管理 |

### 3.6 创建采购申请

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 创建采购申请 |
| 参与者 | 采购员 |
| 前置条件 | 采购员已登录；所需物料已维护。 |
| 后置条件 | 采购申请主表保存为 `DRAFT`；申请明细可继续新增、修改或删除。 |

### 3.7 提交、撤回采购申请

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 提交、撤回采购申请 |
| 参与者 | 采购员 |
| 前置条件 | 采购申请存在，当前用户为申请人，申请已维护至少一条明细。 |
| 后置条件 | 提交后采购申请状态变为 `PENDING_APPROVAL`，审批历史写入 `SUBMIT` 或 `RESUBMIT`；撤回后状态变为 `WITHDRAWN`，审批历史写入 `WITHDRAW`。 |
| 基本流程 | 1. 采购员在草稿或驳回状态点击提交。<br>2. 系统校验申请人、状态和明细。<br>3. 系统更新申请状态和提交时间。<br>4. 系统写入审批历史。<br>5. 如申请处于待审批，采购员可撤回。 |
| 异常流程 | 非申请人操作、无申请明细、状态不允许提交或撤回时返回错误。 |
| 业务规则 | 提交和撤回均在事务中完成，状态变更和审批历史需要保持一致。 |

### 3.8 审批采购申请

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 审批采购申请 |
| 参与者 | 采购经理 |
| 前置条件 | 采购经理已登录；采购申请状态为 `PENDING_APPROVAL`。 |
| 后置条件 | 审批通过后申请状态为 `APPROVED`；审批驳回后申请状态为 `REJECTED`；审批历史记录审批动作、操作人、时间和意见。 |
| 基本流程 | 1. 采购经理进入采购审批页面查询待审批申请。<br>2. 查看申请主信息、申请明细和历史记录。<br>3. 填写审批意见。<br>4. 点击审批通过或驳回。<br>5. 系统更新申请状态并写入审批历史。 |
| 异常流程 | 申请不存在、申请状态不是待审批、审批意见缺失时返回错误。 |
| 业务规则 | 审批结果会同步驱动采购申请状态；已通过申请才允许生成采购订单。 |

### 3.9 创建采购订单

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 创建采购订单 |
| 参与者 | 采购员 |
| 前置条件 | 来源采购申请状态为 `APPROVED`；该申请尚未生成采购订单；供应商状态为 `ACTIVE`；订单明细与申请明细一一对应。 |
| 后置条件 | 采购订单主表状态为 `WAIT_CONFIRM`；订单明细生成并记录单价、金额、累计到货数量和累计入库数量；来源申请回写为 `ORDER_CREATED`。 |
| 基本流程 | 1. 采购员选择已通过采购申请。<br>2. 选择可合作供应商并填写计划交期。<br>3. 系统读取申请明细。<br>4. 采购员为每条明细填写采购单价。<br>5. 系统计算行金额和订单总金额。<br>6. 系统保存订单和订单明细。<br>7. 系统回写采购申请状态。 |
| 异常流程 | 申请不存在、申请未审批通过、供应商不存在、供应商状态非 `ACTIVE`、订单明细为空、明细与申请不一致、同一申请重复生成订单时返回错误。 |
| 业务规则 | 当前设计是一张采购申请对应一张采购订单；订单生成后进入 `WAIT_CONFIRM`，等待供应商确认进入执行阶段。 |

### 3.10 修改、取消、关闭采购订单

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 修改、取消、关闭采购订单 |
| 参与者 | 采购员 |
| 前置条件 | 采购订单存在。 |
| 后置条件 | 待确认订单可修改基本信息和订单明细单价；取消订单回写来源申请为 `APPROVED`；关闭订单状态变为 `CLOSED`。 |
| 基本流程 | 1. 采购员查询订单列表。<br>2. 对 `WAIT_CONFIRM` 订单修改计划交期、备注或明细单价。<br>3. 对 `WAIT_CONFIRM` 订单执行取消。<br>4. 对 `IN_PROGRESS` 或 `PARTIAL_ARRIVAL` 订单填写关闭原因并关闭。 |
| 异常流程 | 订单不存在、订单状态不允许、关闭原因为空时返回错误。 |
| 业务规则 | 后端只允许 `WAIT_CONFIRM` 状态修改订单主信息和明细单价；取消只允许 `WAIT_CONFIRM`。前端当前按钮对未完成/未关闭/未取消订单均展示编辑和取消入口，实际以后端校验为准。 |

### 3.11 供应商确认订单

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 供应商确认订单 |
| 参与者 | 供应商 |
| 前置条件 | 采购订单状态为 `WAIT_CONFIRM`，供应商只能处理与自己绑定的订单。 |
| 后置条件 | 订单状态变为 `IN_PROGRESS`，记录供应商反馈交期、确认时间和反馈说明。 |
| 基本流程 | 1. 供应商查看待确认订单。<br>2. 填写预计交期和反馈说明。<br>3. 点击确认接单。<br>4. 系统回写订单状态和确认信息。 |
| 异常流程 | 订单不存在、非当前供应商订单、订单状态不是待确认时返回错误。 |
| 业务规则 | 本用例是文档和数据字段中的设计目标，但当前前后端尚未实现完整接口。若要保证主链从新订单走通，应优先补齐该状态跳转。 |

### 3.12 到货登记

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 到货登记 |
| 参与者 | 仓库岗 |
| 前置条件 | 采购订单状态为 `IN_PROGRESS` 或 `PARTIAL_ARRIVAL`；仓库状态为 `ENABLED`；订单存在可到货明细。 |
| 后置条件 | 生成到货单和到货明细；订单明细累计到货数量增加；订单状态回写为 `PARTIAL_ARRIVAL`。 |
| 基本流程 | 1. 仓库岗选择采购订单。<br>2. 选择仓库并填写到货日期。<br>3. 按订单明细填写到货数量、合格数量、不合格数量和异常说明。<br>4. 系统校验到货数量不得超过剩余可到货数量。<br>5. 系统生成到货单和到货明细。<br>6. 系统增加订单明细累计到货数量。 |
| 异常流程 | 订单状态不允许、仓库禁用、合格数量与不合格数量之和不等于到货数量、到货数量超过剩余可到货数量、重复选择同一订单明细时返回错误。 |
| 业务规则 | 到货单状态根据是否存在不合格数量或异常说明自动判定为 `NORMAL` 或 `ABNORMAL`；支持分批到货。 |

### 3.13 生成入库单

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 生成入库单 |
| 参与者 | 仓库岗 |
| 前置条件 | 到货单存在且尚未生成入库单；到货明细存在合格数量。 |
| 后置条件 | 生成入库主表和入库明细，入库单状态为 `PENDING`。 |
| 基本流程 | 1. 仓库岗选择到货单。<br>2. 系统读取到货明细。<br>3. 系统按合格数量生成入库明细。<br>4. 系统汇总入库数量并生成入库单号。 |
| 异常流程 | 到货单不存在、该到货单已生成入库单、到货明细为空、无可入库合格数量时返回错误。 |
| 业务规则 | 当前设计是一张到货单只能生成一张入库单；不合格数量不进入入库明细。 |

### 3.14 确认入库与库存更新

| 项目 | 内容 |
| --- | --- |
| 用例名称 | 确认入库与库存更新 |
| 参与者 | 仓库岗 |
| 前置条件 | 入库单状态为 `PENDING`；入库明细存在；来源到货单存在。 |
| 后置条件 | 入库单状态变为 `COMPLETED`；订单明细累计入库数量增加；库存台账增加库存；库存流水写入；采购订单状态更新为 `PARTIAL_ARRIVAL` 或 `COMPLETED`。 |
| 基本流程 | 1. 仓库岗打开待确认入库单。<br>2. 点击确认入库。<br>3. 系统锁定入库单并更新入库状态和入库时间。<br>4. 系统逐条回写采购订单明细累计入库数量。<br>5. 系统按物料和仓库锁定库存台账。<br>6. 库存不存在则新增，存在则增加当前库存。<br>7. 系统写入库存流水，记录变更前、变更数量和变更后数量。<br>8. 系统根据订单明细入库完成情况更新采购订单状态。 |
| 异常流程 | 入库单不存在、状态不允许、来源到货单不存在、入库明细为空、订单明细缺失、库存更新失败或流水写入失败时回滚事务。 |
| 业务规则 | 只有确认入库后才允许改变库存；库存必须由业务单据驱动，不应直接手工修改库存台账。 |

### 3.15 入库取消

| 基本流程 | 1. 仓库岗选择待确认入库单。<br>2. 输入备注或取消原因。<br>3. 系统更新入库单状态为取消。 |
| 异常流程 | 入库单不存在或状态非待确认时返回错误。 |
| 业务规则 | 已确认入库的数据不可取消；如后续需要退货或冲销，应设计独立业务单据。 |

## 四、非功能需求

### 4.1 性能需求

- 用户、物料、仓库、供应商、采购申请、采购订单、到货、入库和操作日志列表均应采用分页查询。
- 采购申请详情、订单详情、到货详情和入库详情应只加载当前单据必要明细，避免一次性加载全量业务数据。
- 常用业务操作，包括提交审批、审批通过、创建采购订单、到货登记、生成入库单和确认入库，应在正常数据量下及时返回明确结果。
- 采购订单、到货、入库和库存更新涉及多表写入，应通过事务保证一致性。

### 4.2 安全需求

- 所有核心业务接口需要登录校验，前端路由按角色限制访问。
- 前端请求应携带 Bearer Token；后端应通过用户上下文识别当前操作人。
- 采购申请提交和撤回应校验当前用户是否为申请人。
- 供应商订单确认和资料维护应限制只能处理自身关联数据；当前供应商侧列表查询仍存在按前端限制的情况，后续应补充后端数据范围校验。
- 管理员重置密码、禁用用户、维护基础资料、审批、订单、到货、入库等关键操作应记录操作日志。

### 4.3 可靠性与数据一致性需求

- 采购申请状态必须按 `DRAFT/REJECTED -> PENDING_APPROVAL -> APPROVED/REJECTED` 规则流转。
- 采购申请提交前必须存在申请明细。
- 已通过申请才能生成采购订单，同一申请只能生成一张采购订单。
- 采购订单必须绑定状态为 `ACTIVE` 的供应商。
- 采购订单明细必须与采购申请明细一致，订单数量来源于申请数量。
- 到货数量不得超过订单明细剩余可到货数量。
- 入库数量来源于到货明细合格数量，不合格数量不得直接入库。
- 一张到货单当前只能生成一张入库单。
- 确认入库需要同步更新入库状态、订单明细入库数量、库存台账、库存流水和采购订单状态，任何一步失败应回滚。
- 库存台账以 `material_id + warehouse_id` 唯一标识，库存流水记录每次业务变更。

### 4.4 易用性需求

- 不同角色只展示与职责相关的页面入口，避免采购员、采购经理、仓库岗之间混淆操作。
- 采购申请页面应清楚展示主表信息、明细、审批历史和可执行动作。
- 采购订单页面应展示订单状态、供应商、计划交期、订单金额、明细到货数量和入库数量。
- 到货页面应区分正常到货和异常到货，并展示合格/不合格数量。
- 入库页面应突出待确认、已完成和已取消状态，避免重复入库。
- 所有状态应使用统一中文标签和颜色标识，前后端状态编码保持一致。

### 4.5 可维护性与扩展性需求

- 后端按 Controller、Service、Mapper 分层，状态流转和事务逻辑应集中在 Service 层。
- 状态编码应集中维护，避免前端和后端各自硬编码后出现不一致。
- 单据编号生成规则应集中封装，避免多个模块重复实现。
- 供应商确认、退货、库存预警、库存查询接口、报表统计可作为后续扩展模块。
- AI 智能助手应只读取或分析主业务数据，不应绕过采购、到货、入库的业务状态机直接改变核心单据。

### 4.6 兼容性需求

- 前端管理端面向桌面浏览器，表格、筛选、弹窗和表单应适配常见桌面分辨率。
- 后端接口统一使用 JSON 请求与响应，分页接口统一返回分页结构。
- 日期、金额和数量字段应保持统一格式：日期使用 `yyyy-MM-dd`，时间使用 `yyyy-MM-dd HH:mm:ss`，数量使用三位小数，金额使用两位小数。
- 文件上传应支持常见资质文件类型，并记录文件名、文件地址、文件大小和 MIME 类型。
